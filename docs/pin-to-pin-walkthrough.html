How the system works, end to end

We follow one real host — a DGA attacker — through every stage: from the packet it sent, to the log Zeek wrote, to the score, to the alert, to the exact moment it is pushed into Elasticsearch and rendered on a dashboard. Every code block and every piece of data below is real, taken from the running system.

TRACING   host = 10.50.0.21   (role: DGA attacker)

The whole pipeline on one line

container sim → tcpdump (labbr0) → Zeek + Suricata → parsers → 8 indicators → UEBA → correlation → explainable alert → Elasticsearch (push) → Kibana

One command runs all of it: make lab-demo. Each stage below names the exact script and shows its real input and output.

Inline capture — tcpdump on the lab bridge

The analysis VM is the gateway for every host, so all traffic crosses one virtual bridge (labbr0). tcpdump records it to a pcap. On the bridge the packet still carries its real source IP (10.50.0.21) — no relabeling.
scripts/lab_capture.shtcpdump -i labbr0 -n -w data/captures/lab/capture.pcap \
        'port 53 or port 443 or port 8443'
↓ capture.pcap (raw packets)

Parse & score — the 8 indicators run

pipeline.build_feature_vectors() reads the Zeek logs, groups records by host, and runs the 8 indicator functions. Each returns a 0–1 sub-score for the host.
src/c2detect/indicators/entropy.py  (real)def subscore(domain, entropy_high=3.5):
    e = longest_label_entropy(domain)         # Shannon entropy of the label
    return max(0.0, min(1.0, e / (entropy_high * 1.5)))

# 'fktqbkqrsqn4920v' is near-random -> high entropy -> sub-score 0.762

Running all 8 indicators for host 10.50.0.21 yields this feature vector (the real numbers):

5

UEBA — how abnormal is this host vs. normal?

OpenUBA is the UEBA engine (running on this VM — no Kubernetes/Spark). The feature vectors are pushed to OpenUBA, which trains on the benign hosts and returns a per-host risk; the adapter calibrates it to a 0–1 anomaly against the benign cohort. (A built-in IsolationForest + z-score is the drop-in fallback via ueba.source.) Host .21 is wildly abnormal (nxdomain_rate 1.0 when benign hosts are ~0), so:
UEBA output for 10.50.0.21  (real, via OpenUBA)anomaly_score = 1.00   (0 = normal, 1 = extreme outlier)
↓ anomaly_score = 1.0
6

Correlation — fuse everything into one confidence

The glass-box engine blends the UEBA anomaly with the weighted indicator bundle, then adds boosts for high-signal combinations. Every term is inspectable.
src/c2detect/correlation/engine.py  (real formula)indicator_component = Σ wᵢ·subscoreᵢ / Σ wᵢ        = 0.497
ueba_component      = ueba_weight · anomaly   = 0.40 · 1.0 = 0.400
base                = ueba_component + (1 - ueba_weight) · indicator_component
                    = 0.400 + 0.60 · 0.497            = 0.698
boost               = +0.10 (dga & nxdomain) +0.10 (entropy & nxdomain) = 0.200
CONFIDENCE          = base + boost = 0.698 + 0.200      = 0.898
Why the boost matters
Two behaviours firing together (DGA structure AND failed lookups) is far stronger evidence than either alone — so the rules add confidence for the combination. This is the correlation that beats single indicators.
↓ confidence = 0.898 (≥ 0.60 threshold → alert)

Push to Elasticsearch — how logs actually land

There are two writers, both talking to Elasticsearch over HTTPS with a password. This is the part you asked about specifically.
Path A — alerts (one document each)
The pipeline’s AlertWriter connects, ensures the index exists with an explicit field mapping, then indexes each alert by its id.
src/c2detect/output/elastic.py  (real)from elasticsearch import Elasticsearch
self.es = Elasticsearch(
    ['https://localhost:9200'],
    basic_auth=('elastic', PASSWORD),   # from ELASTIC_PASSWORD / secrets.env
    verify_certs=False, ssl_show_warn=False)   # self-signed lab cert

def ensure_index(self):                 # create index with typed fields
    if not self.es.indices.exists(index='c2-alerts'):
        self.es.indices.create(index='c2-alerts', mappings=self.MAPPING)

def write(self, alert):                 # <-- the actual push, one doc
    self.es.index(index='c2-alerts', id=alert['alert_id'], document=alert)
Path B — telemetry & scores (bulk)
ingest_es.py pushes thousands of Zeek/Suricata records and per-host scores using the bulk helper (one big request, not one-by-one), tagging each with the host, its label, and a spread @timestamp for the time-series.
scripts/ingest_es.py  (real)from elasticsearch import helpers
docs = []
for r in zeek_parser.read_log('dns.log'):
    docs.append({'_index': 'zeek-dns',
                 '_source': {**r, 'entity': r['id.orig_h'],
                             'label': 'attack', '@timestamp': spread.iso(r['ts'])}})
helpers.bulk(es, docs)                  # <-- the actual push, many docs at once

Kibana — dashboards read the indices

build_dashboards.py creates a data view per index (a pointer + time field) and the dashboard panels. Each panel is just an Elasticsearch aggregation query.

The heatmap reads c2-indicator-scores; the KPI tiles run _count queries; the A/B/C bar reads c2-eval. You see the result at https://172.16.242.14:5601.
